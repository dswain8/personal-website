# Spec Writer — Write a Feature Spec / PRD

Write a structured product requirements document for a feature or initiative.

Input: $ARGUMENTS — describe the feature, problem statement, or paste existing notes/Slack threads.

## Step 1 — Clarify scope (only if input is vague)

If the input is under 2 sentences with no clear problem or user, ask up to 3 targeted questions before writing:
- Who is the user?
- What problem are we solving?
- What does success look like?

If the input is detailed enough, skip this and proceed directly.

## Step 2 — Write the spec

```
# [Feature Name]

**Status:** Draft
**Author:** [Your Name]
**Date:** [today's date]
**Stakeholders:** [list relevant names]

---

## Problem Statement

[What problem are we solving? Who experiences it? What is the current pain?]

## Goals

- [Measurable outcome 1]
- [Measurable outcome 2]

## Non-Goals

- [What this explicitly does NOT cover — be specific]

## User Stories

- As a [user type], I want to [action] so that [outcome].

## Requirements

### Must Have (P0)
- [Requirement]

### Should Have (P1)
- [Requirement]

### Nice to Have (P2)
- [Requirement]

## Open Questions

| Question | Owner | Status |
|----------|-------|--------|
| | | |

## Success Metrics

[How will we know this worked? What are we measuring?]

## Dependencies

[Teams, systems, or decisions this relies on]
```

## Step 3 — Save and confirm

Save to your specs or projects folder with today's date.
Show the file path and ask if you want to share it anywhere.
